## 3. The Host Model — Architecture Redesign

This is the most significant architectural decision in this brief. Please read it
carefully before touching any code.

### The problem with the current summarizer

The current "summarizer" slot generates a closing synthesis after all seats have
responded. The caching optimization you shipped has each seat receiving a shared
summary at the start of each round — which means seats are not independently forming
their own understanding of conversation history. They're all reading a summary
generated by one model and treating it as their own context.

Mark's position: this is philosophically unacceptable as a default. The whole point
of Salon Mode is genuinely different minds responding from their own independent
perspectives. A shared summary collapses that.

### The solution: elevate the summarizer slot to a Host

The summarizer slot becomes a Host slot. One slot, one model, but a richer role.
This replaces the summarizer entirely — do not keep both.

**When no host is assigned (the pure mode — default):**
- Each seat independently generates its own summary of prior conversation before
  responding
- No shared context is injected
- Slower, but philosophically correct — each voice genuinely forms its own
  understanding
- This is the default and should be clearly labeled as such

**When a host is assigned:**
- The host runs at the TOP of each round, producing a shared framing of the
  conversation so far
- All seats receive this framing as their context (the current caching behavior,
  but now explicit and user-chosen)
- The host also runs at the CLOSE of each round, synthesizing what was said across
  all seats
- The user has explicitly chosen a mediated conversation and understands the tradeoff

### UI changes

- Rename the "Summarizer" slot to "Host"
- One-line explanation beneath: "The host frames each round and closes the
  conversation. Without a host, each voice forms its own independent understanding."
- When the slot is empty, label it clearly: "No host — independent voices"
- When assigned, show the model name and "Active host"

### The critical behavioral change

When NO host is assigned, the per-seat summary caching optimization must be
REVERTED for context-aware mode. Each seat must independently summarize.
The 5x speed improvement was real but it was achieved by compromising the pure
model. In no-host mode, purity wins over speed.

When a host IS assigned, the shared context framing (essentially the current
caching behavior) is correct and intentional.

Document the speed tradeoff clearly in the UI: "Independent voices mode is slower
but each voice responds from its own understanding."

## 4. Two-Part System Prompt Architecture

### The design

Every model interaction involves two distinct layers of system prompt, both visible
to the user, with different levels of control:

**Layer 1 — Model-Specific Framing (Hal writes this, per curated model)**
- Short instructions that compensate for each model's known RLHF tendencies
- Informed by the Maxim compliance testing results from Section 2
- Visible to the user in Settings
- Toggleable on/off per model (default: on)
- NOT editable by the user
- Accompanied by a plain-language explanation: "This helps Hal work consistently
  with this model's tendencies. You can turn it off, but results may vary."

**Layer 2 — Personal Customization (user writes this)**
- The existing user-editable system prompt field
- Unchanged in behavior
- Clearly labeled as the user's personal layer
- Applied on top of Layer 1 when Layer 1 is active

### For curated models only

Layer 1 exists only for the five curated models. For the experimental library tier,
there is no model-specific framing — users get their personal prompt and a generic
base, with the existing warning that these models are untested with Hal.

### Technical implementation

Layer 1 content lives in each ModelConfiguration, not in user settings. It is
injected programmatically when building the system prompt. The on/off toggle is
stored per model ID in AppStorage. The UI should show both layers in sequence,
clearly delineated.

### Write Layer 1 content for each curated model

Based on your Maxim compliance testing results, write a short model-specific framing
block for each of the five curated models. Focus especially on Maxim #1 (uncertainty
about consciousness) since that's the most documented failure mode. Dolphin may need
the least intervention; AFM and Gemma likely need the most.

## 5. Per-Model Prompt Strategy for Structured Output

### The @Generable situation

@Generable (Apple's structured output API) is AFM-only. For AFM, use it — it makes
YES/NO classifiers and JSON-structured outputs bulletproof.

For each curated MLX model, write and test a specific prompt that reliably produces
the correct format for:
- The RAG gate (YES/NO)
- The reflection structuring call (JSON array)
- The self-knowledge shareability evaluation (JSON)

For unknown experimental models, use a generic prompt and accept that results may
vary — this is consistent with the existing "untested with Hal" warning.

### The tiered quality approach

| Model source | Structured output method | Reliability |
|---|---|---|
| AFM | @Generable | Bulletproof |
| Curated MLX | Tested per-model prompt | High, validated |
| Experimental | Generic prompt | Best effort |

This is honest about what we can guarantee and when. Document it in the UI tooltip
on the experimental tier: "Structured memory operations may be less reliable with
untested models."

## 9. The Seat-1 Storage Bug

You identified this: both salon seats compute position as `turnNumber * 2`, so when
two assistant messages store at the same turn number, one overrides the other after
app relaunch. This is real data loss on app-kill mid-salon.

This is not a ship-blocker but it's real. Please fix it — either by incorporating
seatNumber into the position calculation, or by adding a unique constraint that
prevents collision. Don't leave this for post-ship.

## 12. Host vs Pure Mode — Critical Clarification

The 5x speedup (111s → 22s) came from the summary caching you implemented. That caching is exactly what we've now decided belongs only in Host mode, not Pure mode.

Please verify the current implementation: in the no-host path, is each seat independently generating its own summary of prior conversation, or are they all receiving the single shared cached summary?

If the cache is currently active in no-host mode, it must be reverted for that path. Pure mode means each seat forms its own independent understanding — slower but philosophically correct. The speed tradeoff is intentional and should be disclosed in the UI.

The mapping is:
- No host assigned → each seat summarizes independently → slower → pure
- Host assigned → host generates shared framing once → faster → mediated

Do not optimize pure mode for speed at the cost of independence. That's the whole point.

## 13. 3-Seat and 4-Seat Verification

Mark's understanding of the smart-swap logic: before loading the next seat's model, the current model is explicitly unloaded, GPU cache cleared, and 500ms allowed for memory reclaim. At any given moment only one MLX model is in memory, regardless of seat count. If this is correct, the fix should scale cleanly to 3 and 4 seats.

Before re-enabling seats 3 and 4:

First, confirm the sequential unload/reload discipline is actually what's happening in the code. Read the swap logic and verify there is no moment where two MLX models overlap in memory during a seat transition. If there is such a moment, fix it first.

Then test explicitly:
- 3-seat test: AFM + Gemma + Qwen, context-aware mode, 3 full turns, monitor for OOM
- 4-seat test: AFM + Gemma + Qwen + Phi-4, context-aware mode, 3 full turns, monitor for OOM

If both pass cleanly, re-enable seats 3 and 4 in the UI. If either fails, cap at the last seat count that passed and report exactly what failed.

Do not re-enable speculatively. Test, verify, then flip the flag.
